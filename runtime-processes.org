#+title: Runtime Processes And Coroutines
#+date: 2026-06-28

* Current State

The firmware has a first cooperative Lisp process runner on CM4:

#+begin_src scheme
(spawn '(begin ...))
(yield)
(sleep-ms 100)
(processes)
(process-poll)
(kill pid)
#+end_src

Processes are stored in a fixed-size runtime table. Each process owns a quoted
top-level form or a quoted =begin= body. The firmware loop polls the table
every 10 ms and runs at most one top-level form per poll. =(yield)= advances
to the next top-level form without delay, and =(sleep-ms n)= parks the process
until the board millisecond counter reaches the wake time.

This is not a full continuation system yet. A single top-level Lisp form still
uses the direct recursive evaluator until it returns, and blocking primitives
such as FAT writes and Wi-Fi SDIO commands still block serial input and network
REPL polling while that primitive is running. The process table is rooted for
mark-sweep GC, so suspended process code, environments, and last values remain
live across collections.

The CM0+ image is a bootloader and handoff shim. The CM4 image owns the Lisp
machine, UART console, FAT operations, Wi-Fi driver, and UDP network REPL.
The evaluator is direct recursive Rust code, so a primitive owns the firmware
loop until it returns. Blocking primitives such as FAT writes and Wi-Fi SDIO
commands therefore block serial input and network REPL polling.

* Usage

Quote the program passed to =spawn= so it is stored as process code instead of
being evaluated immediately:

#+begin_src scheme
(spawn '(begin
          (sleep-ms 1000)
          (+ 40 2)))
(processes)
(process-poll 1)
(kill #x00000001)
#+end_src

The automatic poller normally advances ready processes without manual
=(process-poll)=. Manual polling remains useful in a debugger session or in a
future recovery image with the automatic poller disabled.

* 2026-07-01 Live Validation

The current normal Wi-Fi image, started through FAT =boot.lisp= and controlled
over the UDP REPL, can spawn and complete a sleeping top-level process:

#+begin_src sh
tools/send-net-repl.scm --host 10.0.0.9 --payload-only --read-only \
  '(processes)'
tools/send-net-repl.scm --host 10.0.0.9 --payload-only \
  '(spawn (quote (begin (sleep-ms 1000) (+ 40 2))))'
tools/send-net-repl.scm --host 10.0.0.9 --payload-only --read-only \
  '(processes)'
#+end_src

Observed result:

#+begin_example
=> nil
=> #x00000001
=> (((pid . #x00000001)
        (state . done)
        (steps . #x00000002)
        (error . "")))
#+end_example

* Remaining Architecture

The first scheduler is deliberately above the evaluator. A later process model
should replace direct recursive evaluation with explicit evaluator frames:

#+begin_src scheme
(spawn '(begin ...))
(yield)
(sleep-ms 100)
(processes)
(kill pid)
#+end_src

Each runnable process should get a bounded number of evaluator frames, then
yield to the firmware loop. Firmware services such as UART RX, network REPL
polling, timers, and SD/Wi-Fi progress must get time between Lisp steps.

* Required Runtime Changes

1. Replace direct recursive evaluation with a resumable evaluator frame stack.
2. Add process-friendly I/O wait states beyond sleep, completed, error, and
   killed.
3. Move long FAT and Wi-Fi operations behind request objects or incremental
   poll functions before exposing them as process-friendly Lisp forms.
4. Keep synchronous forms available for diagnostics, but document when they
   block the firmware loop.

* Dual-Core Direction

A later split can put the always-responsive front end on CM0+ and evaluator
work on CM4:

- CM0+: UART RX/TX, network REPL framing, reset-safe command queue, watchdog
  style health checks.
- CM4: Lisp evaluator, heap/GC, FAT, Wi-Fi driver work that needs the larger
  core.
- IPC: fixed-size queues in shared SRAM, explicit message ownership, no shared
  mutable Lisp heap across cores.

This is useful once the single-core cooperative model is clear. Moving the
current direct evaluator to CM0+ would not solve blocking FAT or Wi-Fi calls;
it would only move the blockage to another core unless the I/O boundary is also
made asynchronous.
