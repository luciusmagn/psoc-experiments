#+title: PSoC6 Lisp Machine Quickstart
#+date: 2026-06-28

* Scope

This is the short path for working on the CY8CPROTO-062-4343W Lisp
firmware from this checkout. Keep longer dated experiments in =bringup.org=.

* Build And Flash

Check local tools and resources:

#+begin_src sh
tools/setup-modustoolbox.scm --check
tools/prepare-wifi-resources.scm --check
#+end_src

Build and flash the normal quiet Wi-Fi-capable image:

#+begin_src sh
tools/build-flash-lisp.scm --wifi-firmware --wifi-credentials
#+end_src

Build and flash a recovery image that ignores FAT =boot.lisp= at startup:

#+begin_src sh
tools/build-flash-lisp.scm --wifi-firmware --wifi-credentials --skip-boot-file
#+end_src

Use the recovery image when =boot.lisp= contains a blocking form such as
=(wifi-network-bootstrap)= and the board no longer reaches the console loop.
The recovery image leaves the card contents untouched.

* Consoles

Serial console:

#+begin_src sh
tools/serial-console.scm
#+end_src

That command mirrors bytes into =.local/logs/serial-console.log=. To watch the
console without touching the UART device, use:

#+begin_src sh
tools/serial-console.scm --tail-log
#+end_src

Send one small serial smoke-test form:

#+begin_src sh
tools/send-lisp.scm '(+ 1 2)'
#+end_src

Current UART note: on 2026-07-01, =tools/serial-console.scm= was fixed to set
=CREAD= explicitly before opening the KitProg3 UART, and the firmware now
inserts a small idle gap between transmitted UART bytes. The normal 115200
console can show a clean reset banner through the repo script, but RX is still
not trustworthy: a later =(+ 9 10)= smoke test corrupted even at 1 second byte
pacing. =tools/send-lisp.scm= now keeps the UART writer open for 500 ms after
the final carriage return so the submit byte is less likely to be dropped. Use
UART for reset/status observation and last-ditch recovery, then prefer the UDP
or Telnet network REPL for real interaction once Wi-Fi is up.

Check and switch KitProg3 mode:

#+begin_src sh
.local/fw-loader/fw-loader/bin/fw-loader --device-list
.local/fw-loader/fw-loader/bin/fw-loader --mode kp3-hid
#+end_src

If the console becomes corrupt again, first close other serial readers, reset
the host tty to 115200 raw with the tools, and physically replug the KitProg3
USB connection before changing firmware. Use =kp3-bulk= only when HID mode
causes a specific regression that has been documented.

USB-UART physical-path probe:

#+begin_src sh
tools/build-flash-lisp.scm --uart-pin-probe
stty -F /dev/ttyACM0 9600 cs8 -cstopb -parenb -ixon -ixoff -crtscts raw -echo
timeout 5s cat /dev/ttyACM0
tools/build-flash-lisp.scm --wifi-firmware --wifi-credentials --skip-boot-file
#+end_src

The probe should print =UART-PIN-PROBE P5.1 GPIO 9600 8N1=. Use it only when
the normal console remains corrupt after replugging and resetting the tty.

Network REPL, after Wi-Fi and the UDP service are running:

#+begin_src sh
tools/send-net-repl.scm --host BOARD_IP '(+ 1 2)'
tools/send-net-repl.scm --host BOARD_IP --payload-only '(+ 1 2)'
tools/send-net-repl.scm --host BOARD_IP --read-only '(wifi-net-repl-service status)'
tools/send-net-repl.scm --host BOARD_IP --read-only '(wifi-tcp-repl-service status)'
tools/send-net-repl.scm --host BOARD_IP --read-only '(wifi-demux-status)'
tools/send-net-repl.scm --host BOARD_IP --payload-only --read-only --color '(wifi-net-repl-service status)'
#+end_src

Use =--wait 60= for FAT-backed forms such as =ls=, =cat=, =read-file=,
=save-file=, =append-file=, =save-defs=, and =load=.
Simple FAT status and short file reads usually answer within a few seconds on
the current SD path, but writes and larger reads still need the longer wait.
Use =--payload-only= for a cleaner REPL-like display. Each client invocation
uses its own ignored request/response files under =.local/net-repl/=; use
explicit unique sequences when comparing concurrent UDP requests.
Current firmware can build UDP REPL replies up to 1400 payload bytes.
=tools/send-net-repl.scm= prefers =ncat= when it is installed and falls back to
=nc=. The =ncat= path has validated full 1412-byte =LPS2= datagrams; the GNU
=nc= fallback may capture only 1024 bytes, so keep diagnostics compact or use
=ncat= before depending on larger single replies.
=--read-only= sends an =LPS5= request to current firmware and also applies a
host-side allowlist. It is useful for observer/status commands such as
=(wifi-net-repl-service status)=, =(wifi-link-status)=, =(wifi-lease-status)=,
=(wifi-demux-status)=, =(fat-info)=, =(ls)=, =(cat "boot.lisp")=, and
=(read-file "boot.lisp")=. Treat it as an accidental-write guard, not as
authentication.
The default control transport is still the repo-local framed UDP protocol on
port 4665. The first inbound TCP diagnostic is
=(wifi-tcp-listen-once port [poll-frames])=, which accepts one raw TCP
handshake and immediately sends RST/ACK cleanup. It is a prerequisite for a
Telnet-like console, not the console itself.
=(wifi-tcp-receive-once port [poll-frames])= extends that path by accepting one
connection, receiving one TCP payload frame, reporting a 96-byte preview, and
then sending RST/ACK cleanup.
=(wifi-tcp-repl-once port [poll-frames])= accepts one TCP connection, evaluates
one received Lisp form, sends the pretty-printed result over TCP, and closes.
It is still a one-shot diagnostic driven from the UDP REPL, not a persistent
Telnet service.
=(wifi-tcp-repl-service on 2323 1)= starts the Telnet REPL service. It accepts
one TCP session, sends a prompt, parses Telnet IAC commands, refuses
unsupported options with WONT/DONT, handles NVT CR/LF line endings, evaluates
complete Lisp lines, and keeps the session open until the peer closes. The UDP
REPL and Telnet service poll concurrently through a small HAL demux cache, so
UDP status checks can still work while Telnet is enabled. Use
=(wifi-demux-status)= to inspect cache, drain, and drop counters.

Raw inbound TCP smoke test, with the UDP REPL already running:

#+begin_src sh
(tools/send-net-repl.scm --host BOARD_IP --attempts 1 --wait 60 \
  --payload-only '(wifi-tcp-listen-once 2323 80)' \
  > .local/logs/tcp-listen.out 2>&1) &
sleep 2
nc -vz -w 5 BOARD_IP 2323
wait
cat .local/logs/tcp-listen.out
#+end_src

A successful run has =nc= report the port open and the board report
=(status . ready)=, =(listen.status . ready)=, =(syn-ack.status . ready)=,
=(ack.status . ready)=, and =(reset.status . ready)=.

Raw inbound TCP payload smoke test:

#+begin_src sh
(tools/send-net-repl.scm --host BOARD_IP --attempts 1 --wait 60 \
  --payload-only '(wifi-tcp-receive-once 2323 80)' \
  > .local/logs/tcp-receive.out 2>&1) &
sleep 2
printf 'hello from tcp\n' | ncat -w 5 -q 1 BOARD_IP 2323 || true
wait
cat .local/logs/tcp-receive.out
#+end_src

A successful run reports =(payload.status . ready)=, =(payload.bytes . ...)=,
and a =preview= string matching the payload. The host currently sees a reset
because this diagnostic closes the connection without sending a TCP response.

One-shot TCP REPL smoke test:

#+begin_src sh
(tools/send-net-repl.scm --host BOARD_IP --attempts 1 --wait 60 \
  --payload-only '(wifi-tcp-repl-once 2323 80)' \
  > .local/logs/tcp-repl.out 2>&1) &
sleep 2
printf '(+ 40 2)\n' | ncat -w 8 -q 3 BOARD_IP 2323
wait
cat .local/logs/tcp-repl.out
#+end_src

A successful run prints ==> 42= from =ncat= and the board report includes
=(status . ready)=, =(request.status . ready)=, =(eval.status . ready)=, and
=(reply.status . ready)=. The TCP response uses the same pretty printer as the
UDP REPL.

Telnet REPL service smoke test:

#+begin_src sh
tools/send-net-repl.scm --host BOARD_IP --payload-only \
  '(wifi-tcp-repl-service on 2323 1)'
printf '(+ 7 8)\n' | ncat -w 8 -q 3 BOARD_IP 2323
(sleep 1; printf '(list 1 2 3)\r\n(wifi-tcp-repl-service off)\r\n'; sleep 3) \
  | telnet BOARD_IP 2323
tools/send-net-repl.scm --host BOARD_IP --payload-only '(+ 20 23)'
#+end_src

A successful run prints a =lisp>= prompt, ==> 15= through =ncat=, and
=> (1 2 3)= through stock =telnet=. After Telnet turns the service off, the UDP
check prints ==> 43=. Scripted =telnet= needs stdin to stay open briefly after
connect; an immediate =printf | telnet= can close before the client sends the
Lisp line.

If a Wi-Fi smoke marker or UART status shows a valid lease and an enabled UDP
service but host requests still time out, check host routing and ARP/neighbour
state. On networks that overlap a VPN route, add a temporary host route for the
board IP through the Wi-Fi interface, for example:

#+begin_src sh
ip route replace BOARD_IP/32 dev wlan0 src HOST_WIFI_IP
#+end_src

Keep generated MAC addresses, SSIDs, and passphrases out of committed docs and
logs.

* Processes

The current process runner is cooperative and top-level. It runs at most one
top-level form from one ready process every 10 ms:

#+begin_src scheme
(spawn '(begin
          (sleep-ms 1000)
          (+ 40 2)))
(processes)
#+end_src

Use =yield= or =sleep-ms= inside spawned code. =(processes)= reports each
process slot's state and =last-value=, so it is the current way to inspect a
completed asynchronous result. A long-running single form or blocking primitive
still blocks the firmware loop until it returns.

* Boot File

The current FAT =boot.lisp= starts Wi-Fi and enables the network REPL:

#+begin_src scheme
(begin (wifi-network-bootstrap) (wifi-net-repl-service on))
#+end_src

That is useful when it succeeds, but it is still a blocking startup form.
Use =--skip-boot-file= for recovery or for Wi-Fi debugging that should start
from a responsive UART console.
