#+title: PSoC6 Lisp Machine Quickstart
#+date: 2026-06-28

* Scope

This is the short path for working on the CY8CPROTO-062-4343W Lisp
firmware from this checkout. Keep longer dated experiments in =bringup.org=.

* Build And Flash

Check local tools and resources:

#+begin_src sh
tools/setup-modustoolbox.scm --check
tools/prepare-wifi-resources.scm --check
#+end_src

Build and flash the normal quiet Wi-Fi-capable image:

#+begin_src sh
tools/build-flash-lisp.scm --wifi-firmware --wifi-credentials
#+end_src

Build and flash a recovery image that ignores FAT =boot.lisp= at startup:

#+begin_src sh
tools/build-flash-lisp.scm --wifi-firmware --wifi-credentials --skip-boot-file
#+end_src

Use the recovery image when =boot.lisp= contains a blocking form such as
=(wifi-network-bootstrap)= and the board no longer reaches the console loop.
The recovery image leaves the card contents untouched.

* Consoles

Serial console:

#+begin_src sh
tools/serial-console.scm
#+end_src

That command mirrors bytes into =.local/logs/serial-console.log=. To watch the
console without touching the UART device, use:

#+begin_src sh
tools/serial-console.scm --tail-log
#+end_src

Send one small serial smoke-test form:

#+begin_src sh
tools/send-lisp.scm '(+ 1 2)'
#+end_src

Current UART note: after a host reboot, board replug, and KitProg3 HID-mode
switch on 2026-07-01, the normal 115200 console produced readable reset
banners and clean tiny forms such as =(+ 3 4)=. Longer serial input still
corrupted in live tests, so use UART for smoke tests and recovery, then prefer
the UDP network REPL for real interaction once Wi-Fi is up.

Check and switch KitProg3 mode:

#+begin_src sh
.local/fw-loader/fw-loader/bin/fw-loader --device-list
.local/fw-loader/fw-loader/bin/fw-loader --mode kp3-hid
#+end_src

If the console becomes corrupt again, first close other serial readers, reset
the host tty to 115200 raw with the tools, and physically replug the KitProg3
USB connection before changing firmware. Use =kp3-bulk= only when HID mode
causes a specific regression that has been documented.

USB-UART physical-path probe:

#+begin_src sh
tools/build-flash-lisp.scm --uart-pin-probe
stty -F /dev/ttyACM0 9600 cs8 -cstopb -parenb -ixon -ixoff -crtscts raw -echo
timeout 5s cat /dev/ttyACM0
tools/build-flash-lisp.scm --wifi-firmware --wifi-credentials --skip-boot-file
#+end_src

The probe should print =UART-PIN-PROBE P5.1 GPIO 9600 8N1=. Use it only when
the normal console remains corrupt after replugging and resetting the tty.

Network REPL, after Wi-Fi and the UDP service are running:

#+begin_src sh
tools/send-net-repl.scm --host BOARD_IP '(+ 1 2)'
tools/send-net-repl.scm --host BOARD_IP --payload-only '(+ 1 2)'
tools/send-net-repl.scm --host BOARD_IP --read-only '(wifi-net-repl-service status)'
tools/send-net-repl.scm --host BOARD_IP --payload-only --read-only --color '(wifi-net-repl-service status)'
#+end_src

Use =--wait 60= for FAT-backed forms such as =ls=, =cat=, =read-file=,
=save-file=, =append-file=, =save-defs=, and =load=.
Simple FAT status and short file reads usually answer within a few seconds on
the current SD path, but writes and larger reads still need the longer wait.
Use =--payload-only= for a cleaner REPL-like display. Each client invocation
uses its own ignored request/response files under =.local/net-repl/=; use
explicit unique sequences when comparing concurrent UDP requests.
Current UDP REPL replies carry up to 1400 payload bytes, so larger diagnostics
such as =(thermistor-read)= fit in one unfragmented reply on the current stack.
=--read-only= sends an =LPS5= request to current firmware and also applies a
host-side allowlist. It is useful for observer/status commands such as
=(wifi-net-repl-service status)=, =(wifi-link-status)=, =(wifi-lease-status)=,
=(fat-info)=, =(ls)=, =(cat "boot.lisp")=, and =(read-file "boot.lisp")=.
Treat it as an accidental-write guard, not as authentication.
This is not Telnet or rlogin. The current working transport is the repo-local
framed UDP protocol on port 4665; a Telnet-like TCP console still needs a board
TCP listener and minimal option handling.

If a Wi-Fi smoke marker or UART status shows a valid lease and an enabled UDP
service but host requests still time out, check host routing and ARP/neighbour
state. On networks that overlap a VPN route, add a temporary host route for the
board IP through the Wi-Fi interface, for example:

#+begin_src sh
ip route replace BOARD_IP/32 dev wlan0 src HOST_WIFI_IP
#+end_src

Keep generated MAC addresses, SSIDs, and passphrases out of committed docs and
logs.

* Processes

The current process runner is cooperative and top-level. It runs at most one
top-level form from one ready process every 10 ms:

#+begin_src scheme
(spawn '(begin
          (sleep-ms 1000)
          (+ 40 2)))
(processes)
#+end_src

Use =yield= or =sleep-ms= inside spawned code. A long-running single form or
blocking primitive still blocks the firmware loop until it returns.

* Boot File

The current FAT =boot.lisp= starts Wi-Fi and enables the network REPL:

#+begin_src scheme
(begin (wifi-network-bootstrap) (wifi-net-repl-service on))
#+end_src

That is useful when it succeeds, but it is still a blocking startup form.
Use =--skip-boot-file= for recovery or for Wi-Fi debugging that should start
from a responsive UART console.
