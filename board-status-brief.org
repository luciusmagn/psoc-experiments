#+title: PSoC 6 Lisp Machine Status Brief
#+date: 2026-07-01

* What Works Now

The CY8CPROTO-062-4343W board boots a CM0+ loader and a CM4 Rust firmware
named =lisp-psoc-pc=. The firmware starts quietly, leaves the user LED off
after finite startup/status markers, and exposes a tiny Lisp machine.

Current working capabilities:

- UART banner and prompt over KitProg3 USB serial at =115200 8N1= on
  =/dev/ttyACM0=. Firmware TX pacing can produce a clean reset banner, but
  RX can still corrupt tiny forms even with slow host pacing.
- Lisp forms over the framed UDP network REPL on port =4665=.
- Wi-Fi association, DHCP lease acquisition, router ARP, DNS queries, and a
  small HTTP GET smoke test.
- LED4 milestone pulses during normal startup: one pulse after FAT =boot.lisp=
  read success, two after Wi-Fi bootstrap success, and three after UDP REPL
  service enable. LED4 returns off afterward.
- FAT-backed microSD storage for short source/data files, including =boot.lisp=.
- File forms such as =ls=, =cat=, =read-file=, =read-file-chunk=,
  =save-file=, =append-file=, =save-defs=, and =load= within the current
  small-file limits.
- Cooperative top-level Lisp processes with =spawn=, =yield=, =sleep-ms=,
  =processes=, =process-poll=, and =kill=. Process reports include the last
  evaluated value for completed work.
- Raw inbound TCP diagnostics for handshake and payload receive.
- A one-shot TCP Lisp REPL form, =(wifi-tcp-repl-once port [poll-frames])=,
  which accepts one TCP connection, evaluates one received Lisp form, sends the
  pretty-printed result back over TCP, and closes.
- A Telnet Lisp REPL service, =(wifi-tcp-repl-service on 2323 1)=, with
  persistent TCP session state, IAC command parsing, WONT/DONT option refusal,
  NVT line endings, prompts, and pretty-printed Lisp results.
- A small HAL demux cache that lets the UDP REPL service and Telnet service
  poll concurrently without discarding each other's framed packets, with
  =(wifi-demux-status)= counters for cached, drained, and dropped packets.

* Core Usage

Only the board's cores are split today:

- CM0+: bootloader, reset/setup, CM4 image handoff.
- CM4: Lisp evaluator, heap and GC, UART console, FAT storage, Wi-Fi SDIO
  driver, UDP REPL service, TCP diagnostics, and cooperative process polling.

The CYW4343W Wi-Fi/Bluetooth module is driven over SDIO from CM4. The second
PSoC core is not yet used as a live console or I/O front end after boot. The
preferred next architecture is to move always-responsive I/O, Telnet or stream
framing, and health signalling to CM0+ as soon as the single-core service
boundaries are stable enough to define a small shared-memory IPC contract.

* The Lisp

The language is a small Scheme-like Lisp implemented in Rust for this board.
It has s-expressions, integers, machine words, booleans, symbols, strings,
pairs/lists, globals, closures, recursion, and a pretty printer. Core special
forms include =quote=, =if=, =define=, =lambda=, =begin=, and =let=.

The runtime has a mark-sweep heap and a direct recursive evaluator. That means
one Lisp form runs until it returns; long primitives such as FAT writes or Wi-Fi
SDIO operations still block the firmware loop while they execute. The current
process system schedules only between top-level forms, not inside arbitrary
evaluator frames.

Hardware and network operations are Lisp primitives layered through a Rust
Board trait and HAL modules. Command parsing, evaluation, storage, Wi-Fi, and
board peripherals stay separate so low-level register or packet work does not
leak into Lisp reader/evaluator code.

* Closest Planned Features

Near-term work should focus on making interaction less fragile:

1. Promote the small UDP/TCP REPL demux cache into a board-wide packet
   dispatcher for user network code.
2. Start the CM0+ live-I/O split once the IPC surface is small and explicit.
3. Keep the UART console working as a recovery path and continue reducing byte
   corruption.
4. Add paging or streaming for network REPL responses larger than one UDP
   packet.
5. Replace the current top-level process runner with resumable evaluator frames
   so processes can yield inside longer Lisp computations.
6. Expand FAT loading/saving beyond the current small-file limits.
