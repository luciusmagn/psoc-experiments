#+title: CY8CPROTO-062-4343W bring-up notes
#+date: 2026-06-21

* Goal

Turn the CY8CPROTO-062-4343W PSoC 6 Wi-Fi BT Prototyping Kit into a
small interactive machine:

1. Reliable console over the KitProg3 USB-UART bridge.
2. Tiny Lisp-style REPL on that console.
3. Storage through the microSD slot.
4. Wi-Fi through the onboard CYW4343W module.

The serial console milestone is partially working. UART is useful for reset
banners, small smoke forms, and recovery, but the framed UDP REPL is the
current reliable control path for longer interaction.

* Current known-good state

The board is flashed with:

- CM0+ bootloader: =psoc6-cm0-bootloader=
- CM4 app: =lisp-psoc-pc=
- CM4 app flash base: =0x10008000=
- UART: =SCB5= on =P5.1 TX= and =P5.0 RX=
- Host serial device: =/dev/ttyACM0=
- Baud: =115200 8N1=
- User LED: finite reboot marker at startup, then off; console heartbeat
  disabled by default

The current firmware emits a finite reboot marker on LED4 at startup, then
turns LED4 off and intentionally does not blink the LED at idle. The marker is
short, short, long, short, long. This makes resets visible from across the
room while keeping the board quiet during longer bring-up work. To restore a
visible heartbeat from the console:

=> #t

The red user LED then blinks, which means the CM4 app loop is running.

* Hardware facts

** USB devices

The current preferred KitProg3 mode is =kp3-hid= when UART corruption appears
in =kp3-bulk=. In HID mode the host sees KitProg3 as:

#+begin_example
04b4:f154 Cypress Semiconductor Corp. KitProg3 CMSIS-DAP
#+end_example

Bulk mode has also been observed on this board as =04b4:f155=. HID mode keeps
OpenOCD usable and made the USB-UART bridge materially better in the
2026-07-01 validation below, though not perfect.

The USB-UART bridge appears as:

#+begin_example
/dev/serial/by-id/usb-Cypress_Semiconductor_KitProg3_CMSIS-DAP_1F110B1D03201400-if02 -> ../../ttyACM0
#+end_example

** UART wiring

The kit guide says the KitProg3 USB-UART bridge is hard-wired to:

| Target pin | Function |
|------------+----------|
| P5[0]      | UART RX  |
| P5[1]      | UART TX  |
| P5[2]      | UART RTS |
| P5[3]      | UART CTS |

For this exact board, Zephyr's PSoC6 pinctrl metadata says =P5.0/P5.1
SCB5 UART= uses =HSIOM_SEL_ACT_6=.

This mattered: programming =ACT_9= made SCB5 transmit internally but did
not route TX to the KitProg3 bridge. The working live register value is:

#+begin_example
HSIOM.PRT5.SEL0 = 0x00001212
#+end_example

=0x12= is decimal =18=, the =ACT_6= selector.

** LED

LED4 is on =P13.7= and is active-low. The carrier schematic shows
=VDDIO_0 -> LED4 -> R26 -> P13.7=:

- Drive low: LED on.
- Drive high: LED off.

The initial =led off= confusion came from the idle heartbeat overriding
manual LED commands every 500 ms. The local Zephyr and Tock board
metadata model the LED as active-high, but the schematic and live board
behavior match active-low. Current behavior:

- =(led on)= drives =P13.7= low and disables heartbeat.
- =(led off)= drives =P13.7= high and disables heartbeat.
- =(led toggle)= toggles the manual state and disables heartbeat.
- =(led status)= reports the tracked LED state as a boolean.
- =(heartbeat on)= explicitly restores blinking.
- =(heartbeat off)= disables blinking without changing the current manual
  LED state.

Quiet verified state:

#+begin_example
lisp> (led status)
=> #f

lisp> (heartbeat off)
=> #f

lisp> (reg32 #x40310680)
=> #x0000008f
#+end_example

The low four bits in =GPIO.PRT13.OUT= are the SDHC1 data pins preset
high before being handed to the peripheral. =P13.7= remains high, so
LED4 is still off.

** microSD slot

The microSD socket is on the QSPI flash and microSD card breakaway
section. The schematic maps it to the native SDHC/SDIO pins, not to an
ordinary SPI bus:

| Signal       | PSoC6 pin |
|--------------+-----------|
| SD_CMD       | P12.4     |
| SD_CLK       | P12.5     |
| SD_IO0/DAT0  | P13.0     |
| SD_IO1/DAT1  | P13.1     |
| SD_IO2/DAT2  | P13.2     |
| SD_IO3/DAT3  | P13.3     |
| CD_L detect  | P13.5     |

The populated card-detect path is =CD_L -> R61 -> P13.5=. The alternate
detect path through =R62 -> P12.1= is not loaded. The =_L= suffix matters:
the first live reading with the user's inserted 32GB card measured low.
The next useful manual check is removing the card and running =(sd-status)=
again to confirm the opposite state.

The console firmware configures =P13.5= as high-Z input with input buffer
enabled and exposes a passive diagnostic:

#+begin_example
lisp> (sd-status)
=> ((CD_L . low) (inserted . #t) (GPIO.PRT13.IN . #x0000010f) (GPIO.PRT13.CFG . #x6080eeee))
#+end_example

This only proves the card-detect contact and GPIO input path. Reading
sectors still requires the SDHC1 clock setup and card initialization
sequence.

The firmware now configures the removable-card bus pins for SDHC1 at
boot:

| Signal       | HSIOM selector | Drive mode |
|--------------+----------------+------------|
| P12.4 CMD    | 26             | 0x0e       |
| P12.5 CLK    | 26             | 0x0e       |
| P13.0 DAT0   | 26             | 0x0e       |
| P13.1 DAT1   | 26             | 0x0e       |
| P13.2 DAT2   | 26             | 0x0e       |
| P13.3 DAT3   | 26             | 0x0e       |

=0x0e= is Infineon's =CY_GPIO_DM_STRONG=, strong drive with input buffer
enabled. The PDL SD host notes recommend this mode for DAT pins after
card insertion before calling card initialization. =P13.5= stays as GPIO
card-detect rather than being routed to SDHC1.

Verified after flashing:

#+begin_example
lisp> (sd-pins)
=> ((P12.SEL1 . #x00001a1a) (P13.SEL0 . #x1a1a1a1a) (P12.CFG . #x00ee0000) (P13.CFG . #x6080eeee))
#+end_example

The next diagnostic enables the SDHC0 and SDHC1 wrappers and reads
controller registers without sending card commands:

#+begin_example
lisp> (sdhc-regs)
=> ((SDHC0 (WRAP.CTL . #x80000000) (HOST.VERSION . #x00000005) (CAP1 . #x276c6481) (CAP2 . #x08000077) (PSTATE . #x02070000)) (SDHC1 (WRAP.CTL . #x80000000) (HOST.VERSION . #x00000005) (CAP1 . #x276c6481) (CAP2 . #x08000077) (PSTATE . #x03f70000)) (microSD-pins (P12.SEL1 . #x00001a1a) (P13.SEL0 . #x1a1a1a1a) (P12.CFG . #x00ee0000) (P13.CFG . #x6080eeee)))
#+end_example

This proves the firmware can reach both Synopsys SDHC cores.

- =SDHC0= is the Wi-Fi SDIO controller on =P2.0..P2.7=.
- =SDHC1= is the removable microSD controller on =P12.4=, =P12.5=,
  and =P13.0..P13.3=.
- The GPIO package header for =CY8C624ABZI-S2D44A0= says the SDHC1
  HSIOM selector for those pins is decimal =26=.

Sector I/O still needs clock setup, card initialization commands, and a
block-device layer.

*** =(sd-init)= diagnostic

The =(sd-init)= primitive now performs the first identification-stage SD
sequence on =SDHC1=:

1. Reapply card-detect and SDHC1 pin routing.
2. Enable =SDHC1= and its internal clock.
3. Reset the SDHC core.
4. Configure the host for one-bit identification mode.
5. Enable host card power and a 400 kHz SD clock.
6. Send =CMD0= and =CMD8=.

Live result with the inserted 32 GB card:

#+begin_example
lisp> (sd-init)
=> ((status . cmd8-failed) (CMD8 . #x00000000) (ACMD41.OCR . #x00000000) (attempts . 0) (CLK_CTRL . #x00007d07) (PWR_CTRL . #x00000001) (NORM_INT . #x00000000) (ERR_INT . #x00000000) (PSTATE . #x03f30000) (last-error (code . command-timeout) (NORM_INT . #x00000000) (ERR_INT . #x00000000) (PSTATE . #x03f30000)))
#+end_example

Interpretation: the host reaches 400 kHz clock configuration
=(CLK_CTRL=0x7d07)= and card power enable =(PWR_CTRL=0x01)=, but the card
does not answer =CMD8=. The controller also does not report command error
bits. Next checks should focus on whether the socket is actually powered
and whether =SD_CLK= and =SD_CMD= are reaching the card.

* Firmware notes

** Boot shape

PSoC6 starts the Cortex-M0+ after reset and holds Cortex-M4 in reset.
The =psoc6-cm0-bootloader= image runs first, sets the CM4 vector-table
base to =0x10008000=, and releases CM4.

The CM4 app must be embedded as a raw binary, not as an ELF file:

#+begin_example
psoc6-cm0-bootloader/src/app.bin
#+end_example

If =file psoc6-cm0-bootloader/src/app.bin= reports =ELF=, the bootloader
will read an ELF header where it expects the CM4 stack pointer.

** Console implementation

=lisp-psoc-pc= is intentionally bare and polling:

- no RTOS
- no semihosting
- SCB5 UART, 115200 8N1
- line editing for printable characters, backspace, Ctrl-C, and
  multiline input when list parentheses are still open
- prompt: =lisp>=
- one reader/evaluator path for both Lisp and board diagnostics

Implemented REPL surface:

- reader: symbols, signed decimal integers, =#x= words, quote syntax,
  lists, and dotted pairs
- evaluator: =quote=, =if=, =define=, =lambda=, =begin=, =let=, global
  definitions, lexical closures, and a bounded eval-depth guard
- heap: fixed object table with cons cells, closures, environment cells,
  interned symbols, and mark-sweep GC
- arithmetic and predicates: ~+~, ~-~, ~*~, ~/~, ~mod~, ~=~, ~<~, ~<=~,
  ~>~, ~>=~, =not=, =eq?=, =nil?=, =atom?=, =pair?=, =number?=,
  =symbol?=, =bool?=
- lists: =cons=, =car=, =cdr=, =list=
- board primitives: =led=, =heartbeat=, =button=, =millis=, =reg32=,
  =poke32=, =regs=, =sd-status=, =sd-pins=, =sd-pinmux=, =sd-clock=,
  =sd-init=, =sd-read=, =sd-read0=, =sdhc-regs=, =heap=, =gc=,
  =reboot=

Hardware diagnostics are Lisp forms now, not a separate command parser.
For example:

#+begin_example
lisp> (reg32 #x40310680)
=> #x0000008f

lisp> (heap)
=> (7 377 384 19)
#+end_example

* Wi-Fi milestone

The Wi-Fi device is the onboard CYW4343W, not a simple UART peripheral.
Expect this to need:

- GPIO power/reset control for the Wi-Fi module.
- SDIO or SPI transport bring-up.
- CYW4343W firmware blobs.
- A driver plus network stack.

Pragmatic path:

1. Keep this Rust repo for bare-metal console, storage, and Lisp work.
2. Bring up a known Infineon/ModusToolbox or Zephyr Wi-Fi sample for
   =CY8CPROTO-062-4343W=.
3. Once Wi-Fi works in a known stack, decide whether to port the needed
   pieces into this Rust firmware or move the Lisp machine onto a Zephyr
   firmware base.

* 2026-06-22 SDHC1 diagnostic trace

The active firmware keeps board diagnostics inside the Lisp REPL. There
is no separate command parser for SDHC or register inspection. Useful
forms while keeping the board quiet:

(sd-pins)
(sd-clock)
(sd-init)
(sd-read 0)
(sd-read0)
(sdhc-regs)

The latest SDHC1 test was flashed with heartbeat disabled and LED4 off.
Card detect is working:

#+begin_example
(sd-status)
=> ((CD_L . low) (inserted . #t)
    (GPIO.PRT13.IN . #x0000010f)
    (GPIO.PRT13.CFG . #x6080eeee))
#+end_example

The removable microSD pins are muxed to SDHC1 with HSIOM selector 26 and
strong-input drive on CMD, CLK, and DAT0..3:

#+begin_example
(sd-pins)
=> ((P12.SEL1 . #x00001a1a)
    (P13.SEL0 . #x1a1a1a1a)
    (P12.CFG . #x00ee0000)
    (P13.CFG . #x6080eeee))
#+end_example

=sd-clock= confirms that firmware now enables the SDHC1 high-frequency
root clock from PATH0/FLL with a divide-by-two root divider:

#+begin_example
(sd-clock)
=> ((CLK_PATH0 . #x00000000)
    (CLK_HF0 . #x80000000)
    (CLK_HF2 . #x80000010)
    (FLL_CONFIG . #x810000a3)
    (FLL_CONFIG2 . #x0000000d)
    (FLL_STATUS . #x00000007)
    (selected-HF-Hz . #x02faf080))
#+end_example

=CLK_HF2= is the SDHC1 root on this PSoC6 package. =#x80000010= means
enabled, sourced from PATH0/FLL, and divided by two, so the SDHC1 input
clock is 50 MHz. The 400 kHz identification divider is therefore =#x3e=.

=sd-init= now uses that 50 MHz root, sets the SDHC internal PLL-enable
bit when enabling the card clock, emits a compact report that the serial
bridge can carry reliably, and sends =CMD0= as a normal no-response
command. Infineon's PDL marks reset =CMD0= as =CMD_TYPE=ABORT=, but a live
register experiment on this board showed that =CMD_TYPE=ABORT= asserted
=CMD_INHIBIT= and never raised command-complete or command-error. The same
=CMD0= issued as =CMD_TYPE=NORMAL= completed immediately.

=ACMD41.OCR= has the busy and high-capacity bits set. The next firmware
slice should select the card, configure block length if needed, and read

* 2026-06-22 raw microSD sector 0 read

The next slice added a non-DMA single-block read path and exposes it as
=sd-read=, with =sd-read0= kept as a convenience wrapper for sector zero.
The command sequence is:

1. Re-run the identification path used by =sd-init=.
2. Send =CMD2= to move the card toward standby.
3. Send =CMD3= and retain the returned RCA.
4. Send =CMD7= to select the card.
5. Configure =BLOCKSIZE=512=, =BLOCKCOUNT=1=, and read transfer mode.
6. Send =CMD17= for sector 0 and drain 128 words from =BUF_DATA_R=.

Live sector-zero result:

#+begin_example
(sd-read 0)
=> ((status . ready)
    (init-status . ready-sdhc)
    (sector . #x00000000)
    (RCA . #x00005048)
    (OCR . #xc0ff8000)
    (attempts . 2)
    (CMD17.response . #x00000900)
    (last-error)
    (first-words #x00000000 #x00000000 #x00000000 #x00000000
                 #x00000000 #x00000000 #x00000000 #x00000000)
    (MBR.sig . #x0000aa55)
    (partition0.status . #x00000000)
    (partition0.type . #x00000083)
    (partition0.lba . #x00000800)
    (partition0.sectors . #x039a3800)
    (NORM_INT . #x00000000)
    (ERR_INT . #x00000000)
    (PSTATE . #x03f30000)
    (BLOCK_SIZE . #x00000200)
    (BLOCK_COUNT . #x00000000)
    (XFER_MODE . #x00000012)
    (CMD_R . #x0000113a)
    (ARGUMENT . #x00000000))
#+end_example

=MBR.sig=#xaa55= and partition type =#x83= prove that firmware is reading
actual sector data from the inserted card. The first partition starts at
LBA =#x800= and spans =#x039a3800= sectors. The first eight words are zero
because this card's MBR boot-code area starts with zero-filled bytes.

=sd-read0= returns the same report for sector zero. A nonzero read also
works; this confirms that the Lisp argument is reaching =CMD17=. Reading
the first partition sector:

#+begin_example

    (OCR . #xc0ff8000)
    (attempts . 2)
    (CMD17.response . #x00000900)

                 #x00000000 #x00000000 #x00000000 #x00000000)
    (MBR.sig . #x00000000)
    (partition0.status . #x00000000)
    (partition0.type . #x00000000)
    (partition0.lba . #x00000000)
    (partition0.sectors . #x00000000)
    (NORM_INT . #x00000000)
    (ERR_INT . #x00000000)
    (PSTATE . #x03f30000)
    (BLOCK_SIZE . #x00000200)
    (BLOCK_COUNT . #x00000000)
    (XFER_MODE . #x00000012)
    (CMD_R . #x0000113a)
    (ARGUMENT . #x00000800))
#+end_example

The generic sector report only fills =partition0.*= fields when the read
sector has the MBR signature. This keeps arbitrary sector reads from
displaying random payload bytes as partition metadata.

During this slice, one post-flash read exposed an initialization race: the
driver reset the command line immediately after a successful normal
=CMD0=. That reset was a leftover from the earlier =CMD_TYPE=ABORT=
failure. Removing the post-=CMD0= command-line reset and waiting for
=CMD_COMPLETE= made =sd-init= return =ready-sdhc= again:

* 2026-06-22 Wi-Fi SDIO bring-up

Started the Wi-Fi work as the first explicit HAL split:

- =hal::wifi_sdio= owns CYW4343W power sequencing, P2 SDIO pin muxing,

* 2026-06-22 Firmware stratification

Moved the active firmware toward explicit layers:

1. =hal::console= owns SCB5 UART setup and byte I/O.
2. =hal::board= owns LED/button GPIO, raw register access policy, uptime
   and heartbeat state, and conversion from hardware reports into the
   Lisp-facing board trait.
3. =hal::micro_sd= and =hal::wifi_sdio= own SDHC register sequences for
   the microSD socket and CYW4343W SDIO bus.
4. =lisp_store= stays above the HAL and uses sector reads/writes instead
   of direct SDHC register access.
5. =main= is now startup plus the REPL loop.

This should keep new Wi-Fi and storage diagnostics as Lisp forms built on
HAL operations, without growing command parsing or direct register writes
inside the evaluator.

* 2026-06-22 Wi-Fi backplane setup

* 2026-06-22 Wi-Fi CMD53 read diagnostic

Added =(wifi-cmd53-read function address count)= as a small read-only
diagnostic. The first implementation is deliberately constrained:

1. Function must be 1 through 7.
2. Address must fit the SDIO 17-bit register address field.
3. Count must be 4, 8, 12, or 16 bytes so the SDHC FIFO is read in whole
   words.
4. The form runs =(wifi-setup-backplane)= first, requests ALP through
   function 1 =SDIO_CHIP_CLOCK_CSR=, sends CMD53 byte mode, and reports

* 2026-06-23 Wi-Fi firmware start

Added =(wifi-start-firmware)= as the first one-shot boot operation for the
CYW43430 firmware. The form reuses the same local-resource feature as
=(wifi-load-firmware)= and performs the WHD SDIO sequence in one selected
card session:

1. load and verify =4343WA1.bin= in SOCSRAM;
2. round =wifi_nvram.bin= to four-byte alignment;
3. write NVRAM at =CHIP_RAM_SIZE - 4 - rounded-size=;
4. write the WHD NVRAM size word at =CHIP_RAM_SIZE - 4=;
5. enable SDIO functions 1 and 2;
6. release the WLAN ARM wrapper;
7. wait for =SBSDIO_HT_AVAIL=;
8. configure the SDIO host interrupt masks and function-2 watermark;
9. poll CCCR =IORDY= until function 2 is ready.

* 2026-06-27 Console RX drain and tty no-hup

The REPL main loop used to read at most one UART byte per millisecond. At
115200 baud, a normal Lisp form can arrive much faster than that, so bursts
from =tools/send-lisp.scm= could overflow the SCB5 RX FIFO and leave partial
forms such as =(wifi-w= or damaged symbols at the prompt.

The firmware now drains all currently pending console bytes before sleeping for
the next millisecond tick. The host scripts also force a complete 115200 8N1
mode with no software flow control, no hardware flow control, and =-hupcl= so
one-shot writes do not inherit a fragile tty state.

* 2026-06-28 Wi-Fi DHCP lease acquisition

Added =(wifi-dhcp-acquire)= above the SDPCM data-channel send/receive helpers.
It:

1. Reads =cur_etheraddr= internally without printing raw MAC bytes.
2. Sends DHCP Discover.
3. Polls SDPCM data-channel frames for a matching DHCP Offer.

transaction id, and matches the client hardware address. It extracts DHCP
message type, offered/leased address, server identifier, subnet mask, router,
DNS server, and lease time.

* 2026-06-28 Framed UDP network REPL request

Added =(wifi-net-repl-once)= as the first framed network REPL endpoint. It uses
UDP port =4665= and one request/response frame:

- request: =LPS0=, a big-endian 32-bit sequence number, then one Lisp
  expression.
- response: =LPS1=, the same sequence number, then the pretty-printed result or
  error text.

The HAL owns the peer MAC address internally after parsing a request. Lisp sees
only source IP, source port, sequence number, payload length/hash, response
length/hash, and status fields. Request payloads are capped at 96 bytes to match
the current Lisp string/input size; replies are capped at 512 bytes and report a
truncation flag.

The endpoint accepts unicast packets for the leased IP and limited broadcast
packets, which lets a host discover the board without first reading the serial
console. Replies are unicast to the request source IP/port and the Ethernet
source MAC from the incoming frame.

* 2026-06-28 Network REPL duplicate sequence cache

Added a single-entry response cache in the Lisp machine for the framed UDP REPL.
When a valid request repeats the same source IP, source MAC hash, sequence, and
payload hash as the most recent evaluated request, the firmware reuses the
cached response bytes instead of evaluating the Lisp form again. The reply is
still sent to the current packet's UDP source port, which matters because host
tools such as GNU =nc= may use a fresh local port for each retry.

This is not a full reliability protocol yet. It covers the common retry shape
and prevents duplicate side effects for one outstanding request. A later frame
revision should still add explicit ACK/retry behavior and a response checksum.

* 2026-06-28 Save reloadable Lisp definitions

Added =(save-defs path)= as the simple persistence primitive from the Lisp
machine checklist. It serializes reloadable global definitions into one FAT
source file:

- primitive bindings and built-in constants are skipped silently;
- plain data values are saved as =(define name value)= or
  =(define name (quote value))= when quoting is required;
- top-level closures whose captured environment is =nil= are saved as
  =(define name (lambda ...))=;
- captured closures and other non-reloadable values are counted as skipped;
- output is capped at the 512-byte FAT source-file buffer and reports
  =too-large= instead of writing a truncated file.

The implementation writes the generated source through a single FAT file write
rather than a series of string-sized appends. The earlier multi-chunk version
did eventually complete, but took several minutes and missed the host receive
window. The one-write path returned within the 60-second validation window.

* 2026-06-28 Network REPL ACK frame

Added an optional host-to-board ACK frame for the framed UDP REPL:

- request: =LPS3=, sequence, request checksum, Lisp expression;
- response: =LPS2=, sequence, response checksum, pretty-printed result;
- ACK: =LPS4=, sequence, response checksum.

ACK frames are protocol accounting only. The board parses them in the Wi-Fi
frame path, records =acks-received=, =last.ack.sequence=, and
=last.ack.response-hash= in =(wifi-net-repl-service status)=, and does not
pass the frame to the Lisp reader or evaluator. Older board images ignore the
unknown =LPS4= frame as non-REPL traffic.

=tools/send-net-repl.scm= now sends =LPS4= after it has accepted a verified
response. The existing request retry behavior remains the recovery path; the
single-entry response cache still prevents duplicate side effects for repeated
sequence/hash requests.

* 2026-06-28 Raw TCP SYN probe

Added raw TCP handshake diagnostics as the next step toward =(http-get
"http://...")= and eventually a standard network REPL transport:

- =(wifi-tcp-syn "host" port)= resolves the host with the stored DNS server,
  sends a TCP SYN, parses a SYN-ACK, and sends RST/ACK cleanup;
- =(wifi-tcp-syn-ip #xIPv4 port)= skips DNS and sends the same TCP probe to a
  numeric IPv4 address.

The TCP frame path builds Ethernet, IPv4, and TCP headers directly, including
the TCP IPv4 pseudo-header checksum. It reuses the stored DHCP lease, router
ARP state, local MAC lookup, SDPCM data send path, and F2 frame polling code.

The first direct-IP test exposed a real firmware bug: the TCP frame builder
returned the 60-byte Ethernet minimum frame length, but the buffer was only the
54 bytes needed for Ethernet + IPv4 + TCP headers. Slicing to 60 bytes panicked
and halted the board before the UDP REPL could reply. The fix is to size the
TCP Ethernet frame buffer at =ETHERNET_MIN_FRAME_BYTES=.

* 2026-06-28 Cooperative process runner

Added a first Lisp-level cooperative process runner:

#+begin_src scheme
(spawn '(begin ...))
(yield)
(sleep-ms 100)
(processes)
(process-poll)
(kill pid)
#+end_src

The current scheduler is intentionally top-level. The firmware loop polls the
process table every 10 ms and runs at most one top-level form from one ready
process per poll. =(yield)= advances to the next top-level form, and
=(sleep-ms n)= parks the process until the board millisecond counter reaches
the wake time. A single long-running form still uses the direct recursive
evaluator until it returns, so FAT and Wi-Fi primitives are still synchronous
unless a later slice makes those operations incremental.

The process table is rooted during mark-sweep GC. Suspended process code,
cursor, environment, and last value are marked live so =(gc)= cannot free a
sleeping process out from under the scheduler.

* 2026-06-28 Faster SD/FAT startup

Normal startup depends on reading FAT =boot.lisp= before Wi-Fi starts. The SD
driver re-identifies the card for each sector access, and the identification
path used a fixed one-second bus ramp delay before CMD0. That made FAT-backed
startup and network validation look wedged even when the card and filesystem
were healthy.

Measured before changing the delay, using the validated UDP REPL service image:

#+begin_example
tools/send-net-repl.scm --host 192.168.0.107 --sequence 601 \
  --attempts 1 --wait 60 --payload-only --read-only '(fat-info)'
=> ((status . ready) ...)
elapsed_seconds=63 status=0
#+end_example

The fixed SD bus ramp delay is now 10 ms. This does not make FAT asynchronous;
it only removes the unnecessary one-second wait from every card identification
attempt. Longer-term, SD should keep a selected-card session or expose
incremental process-friendly I/O.

Validation with =--wifi-net-repl-service-boot-smoke --skip-boot-file=:

#+begin_example

=> ((status . ready) ...)

* 2026-06-29 Thermistor read primitive

Added =(thermistor-read)= as the first active thermistor measurement primitive.
The HAL code is isolated in =hal::thermistor= and keeps the low-level pin, SAR,
SARMUX, and SAR-clock setup out of the Lisp evaluator. It powers the RT1
network only around the measurement by driving =P10.0= as thermistor ground and
=P10.3= as thermistor VDD, samples =P10.1= and =P10.2= through the SAR, then
powers the network and SAR path down before returning.

The Lisp report includes status, raw counts, millivolt estimates, poll counts,
and the relevant SAR/PERI/GPIO/HSIOM register snapshots so the first live
hardware readings can be debugged from the REPL instead of requiring a new
firmware image for every register question. The primitive is intentionally not
in the host-side read-only network client allowlist because it drives pins and
changes SAR state.

Validation:

#+begin_src sh

tools/build-lisp.scm --wifi-firmware --wifi-credentials --skip-boot-file
git diff --check
#+end_src

Live validation was added in the 2026-07-01 run below.
