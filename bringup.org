#+title: CY8CPROTO-062-4343W bring-up notes
#+date: 2026-06-21

* Goal

Turn the CY8CPROTO-062-4343W PSoC 6 Wi-Fi BT Prototyping Kit into a
small interactive machine:

1. Reliable console over the KitProg3 USB-UART bridge.
2. Tiny Lisp-style REPL on that console.
3. Storage through the microSD slot.
4. Wi-Fi through the onboard CYW4343W module.

The console milestone is now working.

* Current known-good state

The board is flashed with:

- CM0+ bootloader: =psoc6-cm0-bootloader=
- CM4 app: =lisp-psoc-pc=
- CM4 app flash base: =0x10008000=
- UART: =SCB5= on =P5.1 TX= and =P5.0 RX=
- Host serial device: =/dev/ttyACM0=
- Baud: =115200 8N1=
- User LED default: off; console heartbeat disabled by default

** UART wiring

The kit guide says the KitProg3 USB-UART bridge is hard-wired to:

| Target pin | Function |
|------------+----------|
| P5[0]      | UART RX  |
| P5[1]      | UART TX  |
| P5[2]      | UART RTS |
| P5[3]      | UART CTS |

For this exact board, Zephyr's PSoC6 pinctrl metadata says =P5.0/P5.1
SCB5 UART= uses =HSIOM_SEL_ACT_6=.

This mattered: programming =ACT_9= made SCB5 transmit internally but did
not route TX to the KitProg3 bridge. The working live register value is:

#+begin_example
HSIOM.PRT5.SEL0 = 0x00001212
#+end_example

=0x12= is decimal =18=, the =ACT_6= selector.

** LED

LED4 is on =P13.7= and is active-low. The carrier schematic shows
=VDDIO_0 -> LED4 -> R26 -> P13.7=:

- Drive low: LED on.
- Drive high: LED off.

The initial =led off= confusion came from the idle heartbeat overriding
manual LED commands every 500 ms. The local Zephyr and Tock board
metadata model the LED as active-high, but the schematic and live board
behavior match active-low. Current behavior:

- =(led on)= drives =P13.7= low and disables heartbeat.
- =(led off)= drives =P13.7= high and disables heartbeat.
- =(led toggle)= toggles the manual state and disables heartbeat.
- =(led status)= reports the tracked LED state as a boolean.
- =(heartbeat on)= explicitly restores blinking.
- =(heartbeat off)= disables blinking without changing the current manual
  LED state.

Quiet verified state:

#+begin_example
lisp> (led status)
=> #f

lisp> (heartbeat off)
=> #f

lisp> (reg32 #x40310680)
=> #x0000008f
#+end_example

The low four bits in =GPIO.PRT13.OUT= are the SDHC1 data pins preset
high before being handed to the peripheral. =P13.7= remains high, so
LED4 is still off.

** microSD slot

The microSD socket is on the QSPI flash and microSD card breakaway
section. The schematic maps it to the native SDHC/SDIO pins, not to an
ordinary SPI bus:

| Signal       | PSoC6 pin |
|--------------+-----------|
| SD_CMD       | P12.4     |
| SD_CLK       | P12.5     |
| SD_IO0/DAT0  | P13.0     |
| SD_IO1/DAT1  | P13.1     |
| SD_IO2/DAT2  | P13.2     |
| SD_IO3/DAT3  | P13.3     |
| CD_L detect  | P13.5     |

The populated card-detect path is =CD_L -> R61 -> P13.5=. The alternate
detect path through =R62 -> P12.1= is not loaded. The =_L= suffix matters:
the first live reading with the user's inserted 32GB card measured low.
The next useful manual check is removing the card and running =(sd-status)=
again to confirm the opposite state.

The console firmware configures =P13.5= as high-Z input with input buffer
enabled and exposes a passive diagnostic:

#+begin_example
lisp> (sd-status)
=> ((CD_L . low) (inserted . #t) (GPIO.PRT13.IN . #x0000010f) (GPIO.PRT13.CFG . #x6080eeee))
#+end_example

This only proves the card-detect contact and GPIO input path. Reading
sectors still requires the SDHC1 clock setup and card initialization
sequence.

The firmware now configures the removable-card bus pins for SDHC1 at
boot:

| Signal       | HSIOM selector | Drive mode |
|--------------+----------------+------------|
| P12.4 CMD    | 26             | 0x0e       |
| P12.5 CLK    | 26             | 0x0e       |
| P13.0 DAT0   | 26             | 0x0e       |
| P13.1 DAT1   | 26             | 0x0e       |
| P13.2 DAT2   | 26             | 0x0e       |
| P13.3 DAT3   | 26             | 0x0e       |

=0x0e= is Infineon's =CY_GPIO_DM_STRONG=, strong drive with input buffer
enabled. The PDL SD host notes recommend this mode for DAT pins after
card insertion before calling card initialization. =P13.5= stays as GPIO
card-detect rather than being routed to SDHC1.

Verified after flashing:

#+begin_example
lisp> (sd-pins)
=> ((P12.SEL1 . #x00001a1a) (P13.SEL0 . #x1a1a1a1a) (P12.CFG . #x00ee0000) (P13.CFG . #x6080eeee))
#+end_example

The next diagnostic enables the SDHC0 and SDHC1 wrappers and reads
controller registers without sending card commands:

#+begin_example
lisp> (sdhc-regs)
=> ((SDHC0 (WRAP.CTL . #x80000000) (HOST.VERSION . #x00000005) (CAP1 . #x276c6481) (CAP2 . #x08000077) (PSTATE . #x02070000)) (SDHC1 (WRAP.CTL . #x80000000) (HOST.VERSION . #x00000005) (CAP1 . #x276c6481) (CAP2 . #x08000077) (PSTATE . #x03f70000)) (microSD-pins (P12.SEL1 . #x00001a1a) (P13.SEL0 . #x1a1a1a1a) (P12.CFG . #x00ee0000) (P13.CFG . #x6080eeee)))
#+end_example

This proves the firmware can reach both Synopsys SDHC cores.

- =SDHC0= is the Wi-Fi SDIO controller on =P2.0..P2.7=.
- =SDHC1= is the removable microSD controller on =P12.4=, =P12.5=,
  and =P13.0..P13.3=.
- The GPIO package header for =CY8C624ABZI-S2D44A0= says the SDHC1
  HSIOM selector for those pins is decimal =26=.

Sector I/O still needs clock setup, card initialization commands, and a
block-device layer.

*** =(sd-init)= diagnostic

The =(sd-init)= primitive now performs the first identification-stage SD
sequence on =SDHC1=:

1. Reapply card-detect and SDHC1 pin routing.
2. Enable =SDHC1= and its internal clock.
3. Reset the SDHC core.
4. Configure the host for one-bit identification mode.
5. Enable host card power and a 400 kHz SD clock.
6. Send =CMD0= and =CMD8=.

Live result with the inserted 32 GB card:

#+begin_example
lisp> (sd-init)
=> ((status . cmd8-failed) (CMD8 . #x00000000) (ACMD41.OCR . #x00000000) (attempts . 0) (CLK_CTRL . #x00007d07) (PWR_CTRL . #x00000001) (NORM_INT . #x00000000) (ERR_INT . #x00000000) (PSTATE . #x03f30000) (last-error (code . command-timeout) (NORM_INT . #x00000000) (ERR_INT . #x00000000) (PSTATE . #x03f30000)))
#+end_example

Interpretation: the host reaches 400 kHz clock configuration
=(CLK_CTRL=0x7d07)= and card power enable =(PWR_CTRL=0x01)=, but the card
does not answer =CMD8=. The controller also does not report command error
bits. Next checks should focus on whether the socket is actually powered
and whether =SD_CLK= and =SD_CMD= are reaching the card.

* Firmware notes

** Boot shape

PSoC6 starts the Cortex-M0+ after reset and holds Cortex-M4 in reset.
The =psoc6-cm0-bootloader= image runs first, sets the CM4 vector-table
base to =0x10008000=, and releases CM4.

The CM4 app must be embedded as a raw binary, not as an ELF file:

#+begin_example
psoc6-cm0-bootloader/src/app.bin
#+end_example

If =file psoc6-cm0-bootloader/src/app.bin= reports =ELF=, the bootloader
will read an ELF header where it expects the CM4 stack pointer.

** Console implementation

=lisp-psoc-pc= is intentionally bare and polling:

- no RTOS
- no semihosting
- SCB5 UART, 115200 8N1
- line editing for printable characters, backspace, Ctrl-C, and
  multiline input when list parentheses are still open
- prompt: =lisp>=
- one reader/evaluator path for both Lisp and board diagnostics

Implemented REPL surface:

- reader: symbols, signed decimal integers, =#x= words, quote syntax,
  lists, and dotted pairs
- evaluator: =quote=, =if=, =define=, =lambda=, =begin=, =let=, global
  definitions, lexical closures, and a bounded eval-depth guard
- heap: fixed object table with cons cells, closures, environment cells,
  interned symbols, and mark-sweep GC
- arithmetic and predicates: ~+~, ~-~, ~*~, ~/~, ~mod~, ~=~, ~<~, ~<=~,
  ~>~, ~>=~, =not=, =eq?=, =nil?=, =atom?=, =pair?=, =number?=,
  =symbol?=, =bool?=
- lists: =cons=, =car=, =cdr=, =list=
- board primitives: =led=, =heartbeat=, =button=, =millis=, =reg32=,
  =poke32=, =regs=, =sd-status=, =sd-pins=, =sd-pinmux=, =sd-clock=,
  =sd-init=, =sd-read=, =sd-read0=, =sdhc-regs=, =heap=, =gc=,
  =reboot=

Hardware diagnostics are Lisp forms now, not a separate command parser.
For example:

#+begin_example
lisp> (reg32 #x40310680)
=> #x0000008f

lisp> (heap)
=> (7 377 384 19)
#+end_example

* Wi-Fi milestone

The Wi-Fi device is the onboard CYW4343W, not a simple UART peripheral.
Expect this to need:

- GPIO power/reset control for the Wi-Fi module.
- SDIO or SPI transport bring-up.
- CYW4343W firmware blobs.
- A driver plus network stack.

Pragmatic path:

1. Keep this Rust repo for bare-metal console, storage, and Lisp work.
2. Bring up a known Infineon/ModusToolbox or Zephyr Wi-Fi sample for
   =CY8CPROTO-062-4343W=.
3. Once Wi-Fi works in a known stack, decide whether to port the needed
   pieces into this Rust firmware or move the Lisp machine onto a Zephyr
   firmware base.

The latest SDHC1 test was flashed with heartbeat disabled and LED4 off.
Card detect is working:

#+begin_example

    (GPIO.PRT13.IN . #x0000010f)

The removable microSD pins are muxed to SDHC1 with HSIOM selector 26 and
strong-input drive on CMD, CLK, and DAT0..3:

#+begin_example
(sd-pins)
=> ((P12.SEL1 . #x00001a1a)
    (P13.SEL0 . #x1a1a1a1a)
    (P12.CFG . #x00ee0000)

=sd-clock= confirms that firmware now enables the SDHC1 high-frequency
root clock from PATH0/FLL with a divide-by-two root divider:

#+begin_example
(sd-clock)
=> ((CLK_PATH0 . #x00000000)
    (CLK_HF0 . #x80000000)
    (CLK_HF2 . #x80000010)
    (FLL_CONFIG . #x810000a3)
    (FLL_CONFIG2 . #x0000000d)
    (FLL_STATUS . #x00000007)
    (selected-HF-Hz . #x02faf080))
#+end_example

=CLK_HF2= is the SDHC1 root on this PSoC6 package. =#x80000010= means
enabled, sourced from PATH0/FLL, and divided by two, so the SDHC1 input
clock is 50 MHz. The 400 kHz identification divider is therefore =#x3e=.

=sd-init= now uses that 50 MHz root, sets the SDHC internal PLL-enable
bit when enabling the card clock, emits a compact report that the serial
bridge can carry reliably, and sends =CMD0= as a normal no-response
command. Infineon's PDL marks reset =CMD0= as =CMD_TYPE=ABORT=, but a live
register experiment on this board showed that =CMD_TYPE=ABORT= asserted
=CMD_INHIBIT= and never raised command-complete or command-error. The same
=CMD0= issued as =CMD_TYPE=NORMAL= completed immediately.

* 2026-06-22 raw microSD sector 0 read

The next slice added a non-DMA single-block read path and exposes it as
=sd-read=, with =sd-read0= kept as a convenience wrapper for sector zero.
The command sequence is:

1. Re-run the identification path used by =sd-init=.
2. Send =CMD2= to move the card toward standby.
3. Send =CMD3= and retain the returned RCA.
4. Send =CMD7= to select the card.
5. Configure =BLOCKSIZE=512=, =BLOCKCOUNT=1=, and read transfer mode.
6. Send =CMD17= for sector 0 and drain 128 words from =BUF_DATA_R=.

Live sector-zero result:

#+begin_example
(sd-read 0)
=> ((status . ready)
    (init-status . ready-sdhc)
    (sector . #x00000000)
    (RCA . #x00005048)
    (OCR . #xc0ff8000)
    (attempts . 2)
    (CMD17.response . #x00000900)
    (last-error)
    (first-words #x00000000 #x00000000 #x00000000 #x00000000
                 #x00000000 #x00000000 #x00000000 #x00000000)
    (MBR.sig . #x0000aa55)
    (partition0.status . #x00000000)
    (partition0.type . #x00000083)
    (partition0.lba . #x00000800)
    (partition0.sectors . #x039a3800)
    (NORM_INT . #x00000000)
    (ERR_INT . #x00000000)
    (PSTATE . #x03f30000)
    (BLOCK_SIZE . #x00000200)
    (BLOCK_COUNT . #x00000000)
    (XFER_MODE . #x00000012)
    (CMD_R . #x0000113a)
    (ARGUMENT . #x00000000))
#+end_example

=MBR.sig=#xaa55= and partition type =#x83= prove that firmware is reading
actual sector data from the inserted card. The first partition starts at
LBA =#x800= and spans =#x039a3800= sectors. The first eight words are zero
because this card's MBR boot-code area starts with zero-filled bytes.

=sd-read0= returns the same report for sector zero. A nonzero read also
works; this confirms that the Lisp argument is reaching =CMD17=. Reading
the first partition sector:

#+begin_example
(sd-read #x800)
=> ((status . ready)
    (init-status . ready-sdhc)
    (sector . #x00000800)
    (RCA . #x00005048)
    (OCR . #xc0ff8000)
    (attempts . 2)
    (CMD17.response . #x00000900)
    (last-error)
    (first-words #x00000000 #x00000000 #x00000000 #x00000000
                 #x00000000 #x00000000 #x00000000 #x00000000)
    (MBR.sig . #x00000000)
    (partition0.status . #x00000000)
    (partition0.type . #x00000000)
    (partition0.lba . #x00000000)
    (partition0.sectors . #x00000000)
    (NORM_INT . #x00000000)
    (ERR_INT . #x00000000)
    (PSTATE . #x03f30000)
    (BLOCK_SIZE . #x00000200)
    (BLOCK_COUNT . #x00000000)
    (XFER_MODE . #x00000012)
    (CMD_R . #x0000113a)
    (ARGUMENT . #x00000800))
#+end_example

The generic sector report only fills =partition0.*= fields when the read
sector has the MBR signature. This keeps arbitrary sector reads from
displaying random payload bytes as partition metadata.

During this slice, one post-flash read exposed an initialization race: the
driver reset the command line immediately after a successful normal
=CMD0=. That reset was a leftover from the earlier =CMD_TYPE=ABORT=
failure. Removing the post-=CMD0= command-line reset and waiting for
=CMD_COMPLETE= made =sd-init= return =ready-sdhc= again:

#+begin_example
(sd-init)
=> ((status . ready-sdhc)
    (CMD8 . #x000001aa)
    (CMD8.error)
    (ACMD41.OCR . #xc0ff8000)
    (attempts . 1)
    ...)
#+end_example

* 2026-06-22 Lisp strings

Added bounded string objects to the Lisp heap. Strings are self-evaluating
values, are traced by the existing GC as atom-like heap objects, and print
with escapes. The initial limit is 96 bytes per string; that is enough for
file paths and small console values without introducing a second allocator.

UART smoke test after flashing:

#+begin_example
"boot.lisp"
=> "boot.lisp"

(define path "boot.lisp")
=> path

(string? path)
=> #t

(list "a\n\"b" path)
=> ("a\n\"b" "boot.lisp")
#+end_example

One shell test accidentally sent a literal newline inside a string, and the
reader returned =error: unterminated string=. That is the intended behavior:
literal newline bytes are not allowed in string syntax; use =\n=.

* 2026-06-22 Raw Lisp store

Added a small raw-sector Lisp store so the machine has persistence before
we tackle a real FAT/exFAT layer. This is not a host-readable filesystem.
It uses sector =#x100= as a directory and data sectors beginning at
=#x101=. The store currently holds five named entries, each with content
bounded by the Lisp string limit.

The REPL forms are:

1. =(save-file "name" "one-expression")=
2. =(read-file "name")=
3. =(load "name")=

=load= reads the stored string and evaluates it as one Lisp expression.
Use =(begin ...)= for a multi-form script.

UART smoke test after flashing:

#+begin_example
(save-file "boot.lisp" "(begin (define persisted 41) (+ persisted 1))")
=> ((status . ready)
    (ready . #t)
    (path-len . 9)
    (content-len . 45)
    (directory-sector . #x00000100)
    (data-sector . #x00000101))

(read-file "boot.lisp")
=> ((status . ready)
    (ready . #t)
    (path-len . 9)
    (content-len . 45)
    (directory-sector . #x00000100)
    (data-sector . #x00000101)
    (content . "(begin (define persisted 41) (+ persisted 1))"))

(load "boot.lisp")
=> 42

persisted
=> 41
#+end_example

After =(reboot)=, loading the same path from the card returned =42= again,
which proves the result came from microSD and not from RAM.

The storage layout uses a magic/versioned directory sector, one sector per
entry, and an FNV-1a checksum for the stored content. Invalid or missing
directory data is treated as an empty store on write and =not-found= on
read. This means the earlier fill-pattern test sector was safely replaced
by the store directory during the first =save-file=.

* 2026-06-22 Wi-Fi SDIO bring-up

Started the Wi-Fi work as the first explicit HAL split:

- =hal::wifi_sdio= owns CYW4343W power sequencing, P2 SDIO pin muxing,
  SDHC0 clocking, SDHC0 command traffic, and low-level register snapshots.
- =main.rs= maps HAL reports into Lisp-facing report structs.
- =lisp.rs= exposes the diagnostic as =(wifi-sdio-init)= and prints only
  association lists.

The first SDIO attempts configured P2 correctly but CMD0 timed out. The
diagnostic report showed =SW_RST = #x2= after the timeout, meaning SDHC0's
command reset could not clear. Infineon's HAL source maps SDHC clocks for
this device family as:

| Controller | HF root |
|------------+---------|
| SDHC0      | CLK_HF4 |
| SDHC1      | CLK_HF2 |

The working microSD path had configured =CLK_HF2= for SDHC1. The Wi-Fi path
had not configured =CLK_HF4= for SDHC0. After enabling =CLK_HF4= from
=CLK_PATH0= with divide-by-2, SDIO enumeration worked:

#+begin_example
(wifi-sdio-init)
=> ((status . ready)
    (CMD5.OCR . #xa0ffff00)
    (attempts . 2)
    (RCA . #x00000001)
    (functions . 2)
    (memory-present . #f)
    (last-error)
    (SDHC0
      (WRAP.CTL . #x80000000)
      (GP_OUT . #x0000003b)
      (CLK_CTRL . #x00003e0f)
      (SW_RST . #x00000000)
      (PSTATE . #x03f30000)
      (CMD_R . #x00000703)
      (ARGUMENT . #x00010000)
      (RESP01 . #x00001e00))
    (P2
      (P2.SEL0 . #x1a1a1a1a)
      (P2.SEL1 . #x00001a1a)
      (P2.CFG . #x06eeeeee)
      (P2.OUT . #x00000070)
      (P2.IN . #x0000011f))
    (clock
      (CLK_HF4 . #x80000010)))
#+end_example

The CYW4343W now answers the SDIO card-identification sequence:

1. CMD0 go-idle.
2. CMD5 IO OCR polling.
3. CMD3 RCA assignment.
4. CMD7 card select.

The reported OCR =#xa0ffff00= means the module is an SDIO-only card with
two I/O functions and no memory function. The next Wi-Fi milestone is not
TCP/IP yet; it is CMD52/CMD53 access to CCCR/FBR and then the Broadcom/WHD
mailbox/firmware protocol.

* 2026-06-22 Wi-Fi CMD52 diagnostics

Added Lisp diagnostics for SDIO CMD52 direct byte transfers:

1. =(wifi-cmd52-read function address)=
2. =(wifi-cmd52-write function address byte)=

These forms currently initialize/select the CYW4343W at the start of each
call. That makes individual diagnostics repeatable, but it also means a
separate read after a write does not observe the previous call's enabled
state; the later read starts from a fresh CMD0/CMD5/CMD3/CMD7 sequence.
The next form should be a higher-level operation that enables functions
and polls ready within one selected-card session.

Read-only smoke tests:

* 2026-06-22 Wi-Fi function enable

Added =(wifi-enable-functions mask)=, which performs one selected-card
session:

1. CMD0/CMD5/CMD3/CMD7 SDIO initialization.
2. CMD52 write to CCCR I/O Enable =#x02=.
3. CMD52 polling of CCCR I/O Ready =#x03=.

Function 1 is ready immediately:

#+begin_example
(wifi-enable-functions #x02)
=> ((status . ready)
    (init-status . ready)
    (requested . #x00000002)
    (ready . #x00000002)
    (attempts . 1)
    (write-response . #x00001002)
    (ready-response . #x00001002)
    (last-error)
    ...)
#+end_example

Requesting both functions only brings up function 1 within the current
one-second poll window:

(wifi-enable-functions #x06)
=> ((status . ready-timeout)

    (requested . #x00000006)
    (ready . #x00000002)
    (attempts . 1000)
    (write-response . #x00001006)
    (ready-response . #x00001002)
    (last-error)
    ...)
#+end_example

Function 2 alone does not become ready in the current poll window:

* 2026-06-22 Firmware stratification

Moved the active firmware toward explicit layers:

1. =hal::console= owns SCB5 UART setup and byte I/O.
2. =hal::board= owns LED/button GPIO, raw register access policy, uptime
   and heartbeat state, and conversion from hardware reports into the
   Lisp-facing board trait.
3. =hal::micro_sd= and =hal::wifi_sdio= own SDHC register sequences for
   the microSD socket and CYW4343W SDIO bus.
4. =lisp_store= stays above the HAL and uses sector reads/writes instead
   of direct SDHC register access.
5. =main= is now startup plus the REPL loop.

This should keep new Wi-Fi and storage diagnostics as Lisp forms built on
HAL operations, without growing command parsing or direct register writes
inside the evaluator.

* 2026-06-22 Wi-Fi backplane setup

Added =(wifi-setup-backplane)= as a single selected-card operation modeled
after the early WHD SDIO setup sequence:

1. Initialize and select the SDIO card.
2. Enable function 1 through CCCR IOEN.
3. Switch CCCR BICTRL to 4-bit mode and set the local SDHC0 host width.
4. Set F0/F1/F2 block sizes to 64 bytes.
5. Enable master, function 1, and function 2 SDIO interrupts in CCCR INTEN.
6. Poll CCCR IORDY until function 1 is ready.

Live result after flashing:

#+begin_example
(wifi-setup-backplane)
=> ((status . ready)
    (init-status . ready)
    (IOEN . #x00000002)
    (IORDY . #x00000002)
    (BICTRL.before . #x00000040)
    (BICTRL.after . #x00000042)
    (F0.block-size . #x00000040)
    (F1.block-size . #x00000040)
    (F2.block-size . #x00000040)
    (INTEN . #x00000007)
    (attempts . 1)
    (last-response . #x00001002)
    (last-error)
    (SDHC0 ... (HOST_CTRL1 . #x000000c2) ...))
#+end_example

This confirms the card accepts the WHD-style function 1 setup. The next
meaningful Wi-Fi diagnostic is CMD53 byte-mode access through function 1.
